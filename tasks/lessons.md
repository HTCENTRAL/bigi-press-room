# 教訓・注意事項

## 2026-03-07

### バグ報告の確認手順
- **教訓**: 「重複レコード」の報告を受けた際、すぐにコードのバグを探すより先に「データモデルとして正しい動作ではないか」を確認する
- **ケース**: 1行=1アイテムの仕様で、アイテム2件登録 → 2行生成 → 重複と誤認
- **ルール**: 報告を受けたら「伝票番号が同じか/違うか」「アイテム数はいくつか」を最初に確認する

### コミット対象の確認
- **教訓**: 実装中にファイルを変更するとコミット対象が増える。ユーザーの意図（どこまでをひとつのコミットにするか）を先に確認する
- **ルール**: コミット前に「何をコミットするか」をユーザーに明示して確認を取る

## 2026-03-08

### UXの「状態遷移の明示」原則
- **教訓**: ボタンをdisabledにするだけでは不十分。完了後に「次に何ができるか」を明示しないと非技術者は混乱する
- **ルール**: 完了状態→次アクションへの遷移は、ボタンの表示/非表示と新ボタン提示をセットで設計する
- **具体例**: 登録完了後 [登録する]を非表示 → [伝票を作成・印刷][新規登録]を表示

### GASフロントエンドの冪等性防止はクライアント側で担保する
- **教訓**: GASのサーバー関数は冪等性が保証されないため、二重送信はフロント側で防ぐ必要がある
- **ルール**: フォーム送信完了後は送信トリガーを物理的に除去（display:none）する。disabledは一時的な処理中のみ使用

### resetForm() でカウンターとDOMを必ず同期する
- **教訓**: mediaSetCount/itemCount はグローバル変数。DOM初期化（innerHTML=''）後にカウンターをリセットしないと次のID採番が壊れる
- **ルール**: リセット処理は「DOM消去 → カウンターリセット → 初期要素追加」の順で行う

## 2026-03-09

### 会話のフレッシュ性を保つ運用ルール
- **教訓**: 会話が長くなると古いメッセージが自動圧縮され、細かいニュアンスが失われる
- **自動ロードされるもの**: MEMORY.md（200行上限）/ CLAUDE.md（グローバル・プロジェクト）
- **自動ロードされないもの**: tasks/changelog.md / tasks/lessons.md（明示的に読みに行く必要がある）
- **鉄則**: 「コミット完了 = /clear で会話を切るサイン」
- **重要な教訓は MEMORY.md にも転記すること**（lessons.md は自動ロードされないため）
